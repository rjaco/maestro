# Squad Schema Reference

This document is the authoritative specification for the squad file format. It defines every field, its type, constraints, and whether it is required or optional.

Squad files are Markdown files with YAML frontmatter. The file must be named `<squad-name>.md` where `<squad-name>` is a kebab-case identifier.

---

## Top-Level Frontmatter Fields

### `name`

| | |
|---|---|
| **Type** | string |
| **Required** | Yes |
| **Constraints** | Kebab-case. Must match the filename (without `.md`). No spaces, no uppercase, no underscores. |

```yaml
name: code-review-squad
```

---

### `description`

| | |
|---|---|
| **Type** | string |
| **Required** | Yes |
| **Constraints** | At least 20 characters. Must be substantive — not a placeholder. Must contain at least one verb describing what the squad does. |

```yaml
description: "Coordinates planning, implementation, and review agents to deliver a complete feature end-to-end."
```

---

### `version`

| | |
|---|---|
| **Type** | string |
| **Required** | Yes |
| **Constraints** | Semver format: `MAJOR.MINOR.PATCH`. Pre-release labels allowed (`1.0.0-beta.1`). Build metadata allowed (`1.0.0+build.1`). Must be a string, not an integer. |

```yaml
version: "1.0.0"
```

---

### `author`

| | |
|---|---|
| **Type** | string |
| **Required** | Yes |
| **Constraints** | Non-empty. Must not be a placeholder (`your-github-handle`, `anonymous`, `unknown`). Typically a GitHub handle. |

```yaml
author: octocat
```

---

### `agents`

| | |
|---|---|
| **Type** | list of agent objects |
| **Required** | Yes |
| **Constraints** | At least one entry. Each entry is an agent object (see Agent Object schema below). |

```yaml
agents:
  - role: planner
    agent: maestro-strategist
    model: opus
    focus: "Break down the request into atomic stories"
  - role: implementer
    agent: maestro-implementer
    model: sonnet
    focus: "Implement each story using TDD"
```

---

### `orchestration_mode`

| | |
|---|---|
| **Type** | string (enum) |
| **Required** | No |
| **Default** | `sequential` |
| **Allowed values** | `sequential`, `parallel`, `dynamic` |

| Mode | Behavior |
|------|----------|
| `sequential` | Agents run one after another, each receiving the previous agent's output |
| `parallel` | Agents run concurrently, results merged at the end |
| `dynamic` | The orchestrator decides order and parallelism at runtime based on task structure |

```yaml
orchestration_mode: sequential
```

---

### `shared_context`

| | |
|---|---|
| **Type** | list of strings |
| **Required** | No |
| **Default** | `[]` (empty — no shared context) |
| **Constraints** | Each entry is a relative file path. Must not escape the project root (no `../`, no `/`-prefixed paths, no `~/`, no `$`-prefixed variables). Must not reference sensitive file patterns (see SECURITY.md). |

Each path is resolved relative to the project root at the time the squad runs. If a listed file does not exist when the squad is invoked, the orchestrator will warn and skip that entry (it will not abort).

```yaml
shared_context:
  - docs/architecture.md
  - CLAUDE.md
  - src/types/index.ts
```

---

### `quality_gates`

| | |
|---|---|
| **Type** | list of strings |
| **Required** | No |
| **Default** | `[]` (no gates — squad runs unconditionally) |

Each entry is a plain-English description of a condition that must be true before the squad declares success. Quality gates are evaluated by the QA reviewer agent at the end of the workflow. They are not executable commands — they are assertions that the reviewer agent checks against the produced output.

```yaml
quality_gates:
  - all tests pass
  - no lint errors
  - reviewer agent approved the implementation
  - no TODO comments remain in changed files
```

---

## Agent Object Schema

Each item in the `agents` list is an agent object with these fields.

### `role`

| | |
|---|---|
| **Type** | string |
| **Required** | Yes |
| **Constraints** | Unique within the squad. Describes the agent's function in this specific workflow (e.g., `planner`, `implementer`, `reviewer`). Does not need to match the agent's registered name. |

---

### `agent`

| | |
|---|---|
| **Type** | string |
| **Required** | Conditionally — required unless the entry is a fully defined inline agent |
| **Constraints** | Must be the name of a registered Maestro agent. Built-in values: `maestro-implementer`, `maestro-qa-reviewer`, `maestro-strategist`, `maestro-researcher`, `maestro-fixer`, `maestro-proactive`. |

When `agent` is specified, the orchestrator loads that agent's definition from `agents/`. The `focus` field overrides the agent's default system prompt suffix for this squad.

---

### `model`

| | |
|---|---|
| **Type** | string (enum) |
| **Required** | No |
| **Default** | The registered agent's default model |
| **Allowed values** | `haiku`, `sonnet`, `opus` |

Overrides the model specified in the agent's own definition. Use `opus` for planning and architecture. Use `sonnet` for implementation and review. Use `haiku` for simple, fast tasks.

---

### `focus`

| | |
|---|---|
| **Type** | string |
| **Required** | No (strongly recommended) |
| **Constraints** | Should be specific to this squad's workflow. Appended to the agent's system prompt for this invocation. |

```yaml
focus: "Focus only on the data access layer. Do not modify UI components."
```

---

### `tools`

| | |
|---|---|
| **Type** | list of strings |
| **Required** | No |
| **Default** | The registered agent's default tool set |
| **Constraints** | May only restrict the tool set, not expand it beyond what the registered agent allows. Including `Bash` requires a `justification` field. `dangerouslyDisableSandbox` is never permitted. |

```yaml
tools:
  - Read
  - Grep
  - Glob
```

---

### `justification`

| | |
|---|---|
| **Type** | string |
| **Required** | Required if `tools` includes `Bash` |
| **Constraints** | Must explain why shell access is needed and what commands are expected. Reviewed by maintainers during security review. |

```yaml
tools:
  - Bash
  - Read
justification: "Runs the project's existing test suite via npm test. No other shell commands are issued."
```

---

## Markdown Body

After the closing `---` of the frontmatter, the file must contain a Markdown body with the following sections.

### Required Sections

#### `## Purpose`

What problem this squad solves. When should a user reach for it over a simpler workflow? What kind of tasks is it optimized for?

#### `## Agents`

For each agent in the squad: what it receives as input, what it produces as output, and what decisions it makes.

#### `## Workflow`

A step-by-step description of the agent handoff sequence. For parallel squads, describe what each branch produces and how results are merged.

### Conditionally Required Sections

#### `## Context Sharing`

Required if `shared_context` is non-empty. Explain why each listed file is shared and what the agents do with it.

#### `## Quality Gates`

Required if `quality_gates` is non-empty. Explain how the reviewer agent evaluates each gate.

## Complete Example

```yaml
---
name: full-stack-feature
description: "Delivers a complete feature from planning through implementation and QA review using three coordinated agents."
version: "1.0.0"
author: octocat
agents:
  - role: planner
    agent: maestro-strategist
    model: opus
    focus: "Decompose the feature request into implementable stories with clear acceptance criteria"
  - role: implementer
    agent: maestro-implementer
    model: sonnet
    focus: "Implement each story using TDD. Write tests first."
  - role: reviewer
    agent: maestro-qa-reviewer
    model: sonnet
    focus: "Verify each story's acceptance criteria are met and no regressions were introduced"
orchestration_mode: sequential
shared_context:
  - CLAUDE.md
  - docs/architecture.md
quality_gates:
  - all tests pass
  - no lint errors
  - reviewer approved all stories
---

## Purpose

Use this squad when delivering a feature that requires planning, implementation, and review as distinct steps. It is optimized for features that span multiple files and require coordination between architectural decisions and code-level implementation.

## Agents

- **planner**: receives the user's feature request and produces a list of stories with acceptance criteria
- **implementer**: receives one story at a time and produces tested, working code
- **reviewer**: receives the implementer's output and validates it against the acceptance criteria

## Workflow

1. planner receives the feature request and decomposes it into stories
2. For each story, implementer writes a failing test, then minimal code to pass it, then refactors
3. reviewer checks the implementer's output against the story's acceptance criteria
4. If reviewer rejects, the story is re-dispatched to implementer with the rejection feedback
5. When all stories pass review, the squad declares success

## Context Sharing

- `CLAUDE.md`: shared so all agents understand project conventions and constraints
- `docs/architecture.md`: shared so the planner can produce stories consistent with the existing architecture

## Quality Gates

- **all tests pass**: reviewer runs the test suite and confirms exit code 0
- **no lint errors**: reviewer runs the project linter and confirms no errors
- **reviewer approved all stories**: reviewer explicitly marks each story as accepted before the squad exits
```
