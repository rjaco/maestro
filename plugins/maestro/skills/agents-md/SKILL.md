# agents.md

Generates and maintains an `agents.md` file at the project root that describes Maestro's active agent architecture. External AI tools (Cursor, Windsurf, GitHub Copilot, and others) read `agents.md` to discover available agents, their capabilities, and constraints — the same way they read `CLAUDE.md` for coding conventions.

## Why agents.md

`CLAUDE.md` documents project conventions for humans and AI. `agents.md` documents the AI agents themselves — who they are, what they can do, and what they must not do. The emerging community convention is a machine-readable companion file at the project root.

Cross-tool benefit: an engineer using Cursor alongside Maestro sees the same agent definitions that Maestro's orchestrator uses. No manual synchronization. No duplicate configuration.

## When to Generate

Regenerate `agents.md` whenever the active agent set changes:

| Trigger | Who calls it |
|---------|-------------|
| `/maestro init` | `auto-init` skill, at initialization completion |
| `/maestro squad activate <squad-name>` | `squad` skill, after squad state written to `.maestro/state.local.md` |
| `/maestro agent-teams start` | `agent-teams` skill, after team topology written |
| Post-milestone | `dev-loop` skill, at milestone completion (agent set may have changed) |

## Output Format

Write to `agents.md` at the project root. Overwrite on each generation — the file is always current.

```markdown
# Agents

> Auto-generated by Maestro. Do not edit manually — changes will be overwritten.
> Last updated: <ISO-8601 timestamp>

This file describes the AI agents active in this project. It follows the emerging
`agents.md` community standard for cross-tool agent discovery.

## Built-in Agents

### maestro-orchestrator
- **Description**: Top-level coordinator. Plans milestones, decomposes features into stories, dispatches specialized agents, and runs quality loops.
- **Capabilities**: milestone planning, story decomposition, parallel dispatch, QA evaluation, escalation routing
- **Model**: opus (default), sonnet (checkpoints)
- **Constraints**: never modifies production code directly; delegates all implementation to implementer agents

### maestro-implementer
- **Description**: Implements a single story following TDD. Writes tests first, then code, then refactors.
- **Capabilities**: read, write, edit files; run tests; run build commands
- **Model**: sonnet
- **Constraints**: works in isolated git worktree; does not commit or push; does not modify files outside story scope

### maestro-qa-reviewer
- **Description**: Reviews implementer output against acceptance criteria. Approves or rejects with structured feedback.
- **Capabilities**: read files; run tests; run linter; read audit log
- **Model**: sonnet
- **Constraints**: read-only except for test runs; cannot modify implementation files

### maestro-architect
- **Description**: Designs high-level structure for complex features. Produces ADRs and milestone decompositions.
- **Capabilities**: read all project files; write architecture docs; write story specs
- **Model**: opus
- **Constraints**: does not implement; produces plans and specs only

### maestro-proactive
- **Description**: Lightweight background worker for periodic health checks, dependency audits, and convention drift detection.
- **Capabilities**: read files; run bash commands (read-only); write to .maestro/logs/ and .maestro/notes.md
- **Model**: haiku
- **Constraints**: never modifies project code or config; write access limited to .maestro/ directory

---

## Squad Agents

> No squad currently active. Activate a squad with `/maestro squad activate <name>`.

---

## Runtime Agents

> No agent team currently active.

---

## Constraints Shared by All Agents

- All agents write audit entries to `.maestro/logs/decisions.md` before executing decisions.
- All agents emit telemetry to `.maestro/logs/telemetry.jsonl` after each hook.
- No agent logs file contents, API keys, or user message text.
- Agents do not push to remote repositories.
- Agents do not modify `.maestro/config.yaml` or `CLAUDE.md` at runtime.
```

## Generation Logic

### Step 1: Collect built-in agents

Always include the five built-in agents: `maestro-orchestrator`, `maestro-implementer`, `maestro-qa-reviewer`, `maestro-architect`, `maestro-proactive`. Their definitions are static — copy from the template above.

### Step 2: Collect active squad agents

Read `.maestro/state.local.md`. If `squad_active: true` is present, read `.maestro/squads/<squad-name>.yaml` and extract each agent's role, model, description, and capabilities. Format as a subsection under `## Squad Agents`.

If no squad is active, emit the `> No squad currently active.` placeholder.

### Step 3: Collect runtime team agents

Read `.maestro/team-state.md` if it exists and `status: in_progress`. Extract each agent from the `## Agent Status` table (role, subagent_type, assigned task). Format as a subsection under `## Runtime Agents`.

If no team is active, emit the `> No agent team currently active.` placeholder.

### Step 4: Write the file

Assemble all sections and overwrite `agents.md` at the project root. Update the `Last updated` timestamp to the current ISO-8601 time.

Do not write to any other location. Do not create a copy in `.maestro/`.

## Per-Agent Entry Format

Each agent entry follows this structure:

```markdown
### <agent-id>
- **Description**: <one sentence — what the agent does>
- **Capabilities**: <comma-separated list of capabilities>
- **Model**: <haiku | sonnet | opus> (<qualifier if relevant>)
- **Constraints**: <comma-separated list of hard limits>
```

Keep descriptions to one sentence. Keep constraints to observable limits, not implementation notes.

## Squad Agent Section Format

When a squad is active, replace the `## Squad Agents` placeholder with:

```markdown
## Squad Agents

Active squad: **<squad-name>**

### <squad-role-1>
- **Description**: <description from squad YAML>
- **Capabilities**: <capabilities>
- **Model**: <model>
- **Constraints**: <constraints>

### <squad-role-2>
...
```

## Runtime Team Agent Section Format

When an agent team is running, replace the `## Runtime Agents` placeholder with:

```markdown
## Runtime Agents

Active team: **<team name from team-state.md frontmatter>** (topology: <team_topology>)

| Agent | Role | Model | Current Task |
|-------|------|-------|--------------|
| worker-1 | implementer | sonnet | T02 — Build PriceTable component |
| worker-2 | implementer | sonnet | T03 — Write unit tests |
| reviewer | qa-reviewer | sonnet | idle |
| lead | architect | opus | coordinating |
```

## Regeneration Protocol

```
READ .maestro/state.local.md
  → extract: squad_active, squad_name

READ .maestro/team-state.md (if exists)
  → extract: milestone, team_topology, agent status table, status

ASSEMBLE agents.md:
  1. Header block (auto-generated warning + timestamp)
  2. ## Built-in Agents (always from static template)
  3. ## Squad Agents (from squad YAML if active, else placeholder)
  4. ## Runtime Agents (from team-state.md if in_progress, else placeholder)
  5. ## Constraints Shared by All Agents (always static)

WRITE agents.md at project root
LOG "agents.md updated" to .maestro/logs/telemetry.jsonl
  event: "agents_md_generated"
  context: { squad_active: bool, team_active: bool, agent_count: N }
```

## Integration Points

| Skill | Integration |
|-------|-------------|
| **auto-init** | Calls this skill at the end of `/maestro init` to generate the initial `agents.md`. |
| **squad** | Calls this skill after `squad activate` completes and after `squad deactivate` clears the squad state. |
| **agent-teams** | Calls this skill after team start and after milestone complete. |
| **dev-loop** | Calls this skill at milestone completion (post-milestone trigger). |
| **telemetry** | Generation events are logged to `telemetry.jsonl` as `agents_md_generated` entries. |
| **hooks-integration** | `InstructionsLoaded` hook can trigger regeneration if squad state has changed since last load. |

## Cross-Tool Compatibility

Tools known to read `agents.md` at the project root:

| Tool | How it uses agents.md |
|------|-----------------------|
| Cursor | Displays agent list in the context panel; respects constraint declarations |
| Windsurf | Auto-discovers agents for multi-agent cascade tasks |
| GitHub Copilot | Uses agent descriptions for `@agent` completions in chat |
| Other tools | Any tool following the emerging agents.md community standard |

The format is intentionally human-readable markdown. Tools that parse it do so with flexible heuristics — there is no strict schema requirement. The `### <agent-id>` heading and bullet-list fields (`Description`, `Capabilities`, `Model`, `Constraints`) are the minimal signals most tools look for.
