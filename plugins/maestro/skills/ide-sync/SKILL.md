# IDE Sync

Maestro's agents and skills are defined once in canonical markdown format (CLAUDE.md, SKILL.md files, squad definitions). This skill generates IDE-specific copies so that Cursor, Gemini CLI, GitHub Copilot, and other tools see consistent, up-to-date instructions without manual maintenance.

On init, all copies are generated. On demand, `/maestro sync-ide` regenerates them. Between syncs, drift detection warns when the canonical source has changed but a copy has not been updated.

## Source of Truth

The canonical source is Maestro's own configuration, not the generated copies:

| Canonical Source | What It Defines |
|-----------------|-----------------|
| `CLAUDE.md` | Project conventions, tech stack, workflow rules |
| `skills/*/SKILL.md` | Skill behaviors and commands |
| `squads/<name>/squad.md` | Squad agent definitions |
| `.maestro/config.yaml` | Active feature flags and preferences |
| `.maestro/dna.md` | Project DNA (language, framework, architecture) |

Never edit the generated copies manually — they will be overwritten on the next sync. The canonical sources are the only files that should be modified.

## Generated Targets

### `.cursorrules`

Cursor reads `.cursorrules` at the project root as always-on prompt injection. It does not support dynamic skill invocation — all rules must be present at session start.

Content strategy: extract the most important behavioral constraints from CLAUDE.md and the active squad definition. Translate hooks-based guards into explicit prohibitions.

Format:

```
# Maestro — Cursor Rules
# Auto-generated by ide-sync. Do not edit manually.
# Source: CLAUDE.md + .maestro/config.yaml
# Updated: <ISO-8601 timestamp>

## Project: <project name from dna.md>
<tech stack summary: language, framework, package manager>

## Branch Policy
Never commit to main or master. Always work on a feature branch.
Before any git operation, run `git branch --show-current` to confirm you are not on main.

## Dev Loop
1. Write a failing test first (TDD).
2. Write the minimal code to make it pass.
3. Refactor. Run tests again.
4. Report STATUS: DONE with test count and files changed.

## Conventions
<extracted from CLAUDE.md: naming conventions, file structure, coding standards>

## Active Squad: <squad name or "none">
<if squad active: role assignments and focus areas>

## Constraints
<extracted from CLAUDE.md sensitive areas and quality gates>
```

### `GEMINI.md`

Gemini CLI reads `GEMINI.md` at session start. Supports `@file` includes for referencing external files. No dynamic skill invocation — instructions are static.

Content strategy: more verbose than `.cursorrules`. Include full dev loop steps, workarounds for missing Gemini features (no PreToolUse, no AskUserQuestion), and `@` references for large context sections.

Format:

```markdown
# Maestro — Gemini CLI Instructions
<!-- Auto-generated by ide-sync. Do not edit manually. -->
<!-- Source: CLAUDE.md + skills/ + .maestro/config.yaml -->
<!-- Updated: <ISO-8601 timestamp> -->

## Project
<tech stack summary from dna.md>

## Branch Policy
Always work on a feature branch. Never commit to main.
Before any git operation: run `git branch --show-current` and verify you are not on main.

## Dev Loop

Implement → test → report status.

1. Read the task specification carefully.
2. Write a failing test for the first acceptance criterion.
3. Write code to make the test pass. Minimal, no gold-plating.
4. Refactor without changing behavior. Run tests again.
5. Repeat for each acceptance criterion.
6. Report STATUS: DONE or STATUS: BLOCKED.

## Approval Checkpoints
When you need user approval before proceeding, print exactly:
  CHECKPOINT: [description of completed work]
  OPTIONS: Continue / Abort / Modify
Then stop. Do not continue until the next message includes a choice.

## Context
@.maestro/dna.md

## Conventions
<extracted from CLAUDE.md>

## Notes on Maestro Feature Gaps
- No PreToolUse hooks: branch guard is enforced by the instruction above, not automatically.
- No AskUserQuestion: use CHECKPOINT pattern above for interactive decisions.
- No parallel agents: run implementer phase, then QA review phase, sequentially in one session.
```

### `agents.md`

Cross-tool agent discovery file. Delegated entirely to the `agents-md` skill — ide-sync calls that skill to generate or refresh `agents.md` rather than duplicating its logic.

On sync, call `agents-md` regeneration. Do not write `agents.md` directly.

Reference: `skills/agents-md/SKILL.md`

### `.github/copilot-instructions.md`

GitHub Copilot reads `.github/copilot-instructions.md` for repository-level instructions. Applied to Copilot Chat and Copilot in the editor.

Content strategy: concise. Copilot instructions are injected into every prompt, so brevity matters. Focus on constraints and the expected output shape.

Format:

```markdown
# Maestro — Copilot Instructions
<!-- Auto-generated by ide-sync. Do not edit manually. -->
<!-- Source: CLAUDE.md + .maestro/config.yaml -->
<!-- Updated: <ISO-8601 timestamp> -->

## Project
<one-line: language, framework, package manager>

## Branch Policy
Never commit to main. Use feature branches.

## Code Style
<top 5 conventions from CLAUDE.md, one line each>

## Testing
- Framework: <test runner from dna.md>
- Run: <test command>
- Write tests before implementation (TDD).

## Completion Format
When asked to implement a feature, end your response with:
  STATUS: DONE — <N> tests passing, files: <list>
```

## Format Translation Rules

Each IDE has constraints that require content transformation, not just copy-paste.

| Canonical Feature | `.cursorrules` | `GEMINI.md` | `copilot-instructions.md` |
|-------------------|---------------|-------------|--------------------------|
| Skill invocation (`/maestro X`) | Omit — not supported | Omit — not supported | Omit — not supported |
| `AskUserQuestion` checkpoint | Replace with "stop and print CHECKPOINT:" | Replace with CHECKPOINT pattern | Omit — implicit in conversation |
| Hook-based branch guard | Convert to prohibitive instruction | Convert to prohibitive instruction | Convert to single-line rule |
| Parallel agent dispatch | Omit — not available | Sequential phases workaround | Omit |
| Model selection rules | Omit — no model control | Omit — no model control | Omit |
| Agent dispatch table | Summarize roles only | Summarize roles only | Omit |
| Quality gates list | Include as checklist | Include as checklist | Include as checklist |

### Omit vs. Replace

**Omit** when the feature has no analogue and including a broken reference would confuse the IDE agent.

**Replace** when the underlying intent can be preserved through a workaround instruction. Always prefer replace over omit when the constraint is a hard requirement (e.g., "never commit to main").

## Drift Detection

Compare each generated file against its canonical source to detect staleness.

### Drift Check Algorithm

1. Read the `Updated:` timestamp from the header of each generated file.
2. Compare against the modification time of each canonical source that contributed to it.
3. If any canonical source is newer than the generated file's `Updated:` timestamp, mark the target as drifted.

```
+----------------------------+
| Canonical source mtime     |  > generated file timestamp?
+----------------------------+
          |
        drifted?
        /      \
      Yes        No
       |          |
  warn user    no action
```

### Drift Report

When drift is detected (on session start or on explicit check):

```
[ide-sync] Drift detected — canonical sources have changed since last sync.

  Drifted targets:
    .cursorrules             (CLAUDE.md modified 2026-03-18, synced 2026-03-15)
    GEMINI.md                (squads/full-stack-dev/squad.md modified 2026-03-17, synced 2026-03-15)

  Unchanged:
    .github/copilot-instructions.md
    agents.md

  Run /maestro sync-ide to regenerate all drifted targets.
```

The drift warning is non-blocking. The IDE copies remain usable — they are just out of date. Maestro does not automatically regenerate on drift; it requires an explicit `/maestro sync-ide` command.

### Drift Check Triggers

| Trigger | Action |
|---------|--------|
| Session start | Silent drift check. Warn if any target is drifted. |
| After `/maestro init` | Auto-sync all targets (no drift check needed — always regenerate). |
| After `squad activate` / `squad deactivate` | Regenerate `.cursorrules` and `GEMINI.md` (squad section changed). |
| After `squad create` | Regenerate all targets. |
| Manual `/maestro sync-ide` | Full regeneration of all targets. |

## Operations

### `/maestro sync-ide`

Regenerate all IDE-specific copies from canonical sources.

1. Read canonical sources: `CLAUDE.md`, `skills/*/SKILL.md` (relevant skills), active squad from `.maestro/state.local.md`, `.maestro/dna.md`, `.maestro/config.yaml`.
2. Apply format translation rules for each target.
3. Write each target file, overwriting any existing content.
4. For `agents.md`: call the `agents-md` skill regeneration instead of writing directly.
5. Update the `Updated:` timestamp in each file's header to the current ISO-8601 time.
6. Report:

```
+---------------------------------------------+
| IDE Sync Complete                           |
+---------------------------------------------+

  .cursorrules                  regenerated
  GEMINI.md                     regenerated
  agents.md                     regenerated (via agents-md skill)
  .github/copilot-instructions.md  regenerated

  Source: CLAUDE.md + .maestro/dna.md + squad: full-stack-dev
```

### `/maestro sync-ide --target <name>`

Regenerate a single target. Valid values: `cursorrules`, `gemini`, `agents`, `copilot`.

Example: `/maestro sync-ide --target gemini` regenerates only `GEMINI.md`.

### `/maestro sync-ide --check`

Run drift detection only. Do not regenerate. Print the drift report and exit.

## Init Behavior

When `/maestro init` completes, run a full sync automatically:

1. Generate all four targets.
2. Create `.github/` directory if it does not exist.
3. Log the sync result at the end of the init summary:

```
IDE sync:
  .cursorrules                    created
  GEMINI.md                       created
  agents.md                       created
  .github/copilot-instructions.md created
```

If `CLAUDE.md` does not exist yet (pre-init state), skip sync silently. Sync will be triggered once init completes and `CLAUDE.md` is written.

## File Ownership

Each generated file includes a header comment marking it as auto-generated:

```
# Auto-generated by Maestro ide-sync. Do not edit manually.
# Source: CLAUDE.md + .maestro/config.yaml
# Regenerate with: /maestro sync-ide
```

If a user has manually edited a generated file (detected by content that doesn't match what would be generated from the canonical source), warn before overwriting:

```
[ide-sync] Warning: .cursorrules appears to have been manually edited.

  Regenerating will overwrite your changes.
  To preserve them, move them to CLAUDE.md and run /maestro sync-ide.

  Proceed? [yes / no]
```

Manual edit detection heuristic: if the `Updated:` header line is missing or has been removed, treat the file as manually edited.

---

## State Schema

No persistent state file. Sync state is derived from file timestamps at drift-check time.

The `Updated:` header in each generated file is the only persisted sync state. It records when Maestro last wrote the file — used as the baseline for drift comparison against canonical source mtimes.
